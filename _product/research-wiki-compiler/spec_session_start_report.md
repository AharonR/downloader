# Session-Start Report Specification

Source: `h13_design_constraints.md` (Constraint 8), `h13_pilot_learnings.md` (Bottleneck 2: Observability).

## Why This Exists

The pilot wiki (this project's memory system, 2 weeks of operation) revealed that without observability, the user feels "blind" to the wiki's state. Trust erodes not because the wiki is wrong, but because the user can't tell what's in it without reading every file. This is the mechanism behind PKM abandonment.

**The observability gap made the compile gap invisible**: there was no signal showing what was sitting in the raw queue. If a report said "3 session notes not yet compiled," the user would know to trigger compilation. Without it, raw notes are invisible — so compile failures are also invisible until retrieval fails.

## When It Runs

Automatically, at the start of every session — before any query. The report is the first thing the user sees.

## Seven Signals

### 1. Session Digest
**Question answered**: What changed since I was last here?
**Content**: List of entity pages created, updated, or merged since last session. Includes: page name, change type (created/updated/merged), source papers.

### 2. Raw Queue
**Question answered**: What has NOT been compiled yet?
**Content**: Papers annotated in Zotero but not yet run through the compile phase. Count + list. This is the most critical signal — it makes deferred compilation visible.

### 3. Coverage Heat Map
**Question answered**: Where is my wiki rich vs. thin?
**Content**: Concepts ranked by coverage indicator (high/medium/low/stub). High = 3+ source papers + annotations. Stub = single mention, no dedicated entity page.

### 4. Staleness Report
**Question answered**: What might be outdated?
**Content**: Entity pages not updated in >30 days while new papers touching the same concepts have been added. Flags potential obsolescence.

### 5. Open Questions Log
**Question answered**: What did the lint phase flag that I haven't resolved?
**Content**: Questions generated by the lint phase that don't have wiki answers yet. Examples: "Paper A and Paper B make conflicting claims about X — which is correct in your assessment?"

### 6. Contradiction Log
**Question answered**: Where does my library disagree with itself?
**Content**: Detected contradictions between entity page claims. Each entry: claim A (source) vs claim B (source), detected date, resolution status (unresolved/resolved/acknowledged).

### 7. Index Health
**Question answered**: Is the wiki structurally sound?
**Content**: Files not in the index, broken backlinks, orphan pages (no inbound links), entity pages with no source papers.

## Output Format

```markdown
# Wiki Status — 2026-04-13

## Since Last Session (2026-04-11)

- Updated: `concepts/causal-inference.md` (added evidence from papers/angrist-2024)

## Raw Queue (3 papers)

```

## Design Notes

- The report must be generated automatically, not on request. If the user has to remember to ask, it defaults to deferral.
- Keep it concise — the full report should fit in one screen (~30 lines). Expand only on request.
- The raw queue is the highest-value signal. If nothing else ships, ship the raw queue count.
- The contradiction log becomes significantly richer once H19 (argument graph) is built — typed edges make contradictions queryable, not just flaggable.
